U1

2-Watt Stereo Audio Amplifier PCB

CMPE 110 · Full PCB lifecycle, from LTSpice to a soldered board

KiCad LTSpice LM386 Team of 3

Designed, simulated, and manufactured a two-channel audio amplifier PCB built around the LM386 low-voltage op-amp, taking it from a theoretical circuit to a working, soldered board with measured output.

RoleCircuit design, schematic capture, layout, and assembly (team of 3, with Alex Chavez and Zemarai Rokai)
StackLTSpice simulation → KiCad schematic capture → footprint assignment → PCB layout & routing → Gerber/BOM export
VerificationElectrical Rules Check (0 errors/warnings), Gerber viewer inspection, oscilloscope waveform capture at 15/20/25 kHz
✓ Amplified audio output confirmed on both channels ✓ Clean 0-error schematic & DRC pass ⚠ Learned layer/via management the hard way

Design & Simulation

  • Built the LM386-based gain stage in LTSpice with a potentiometer acting as a volume control, verifying clean amplification with no clipping before touching KiCad.

Schematic Capture

  • Selected 470 µF filtering caps and 0.047 µF bypass caps based on datasheet values to keep the signal clean.
  • Resolved initial pin conflicts to bring the Electrical Rules Check down to 0 errors and 0 warnings.

Footprint Assignment & Layout

  • Matched DIP-8 through-hole footprints to the LM386 and radial capacitor footprints to measured part diameters.
  • Untangled a messy initial "rat's nest" placement and learned when to use vias to jump between top and bottom copper layers — the most challenging part of the project as a team new to two-layer boards.

Manufacturing & Assembly

  • Generated Gerber, drill, and BOM files, then inspected them in GerbView before submission.
  • Hit real-world soldering issues — bridged and cold joints requiring rework — which stressed some pads, a hands-on lesson that a good design file is only half the battle.
  • Verified power delivery via the onboard LED, then confirmed amplified audio output by ear and on an oscilloscope across multiple frequencies.

U2

Coin-Operated Vending Machine (FSM)

CMPE 125 · Moore machine design in Verilog on a Basys 3 FPGA

Verilog FSM Design Basys 3 Testbench

Designed a Moore-machine finite state machine that tracks inserted coins (nickels, dimes, quarters) toward a 25-cent soda, dispensing and returning correct change across 10 reachable states — implemented and verified on real FPGA hardware.

Logic10-state Moore FSM (S0–S9) tracking running coin totals in 5¢ increments, with dedicated dispense states for exact and over-payment
Inputs / OutputsDebounced, edge-detected coin switches (nickel/dime/quarter) → dispense, returnNickel, returnDime, returnTwoDimes signals + 4-bit state LEDs
VerificationCustom testbench with binary test-vector file covering 10 payment scenarios, plus live hardware validation on the Basys 3
✓ Verified correct change across all coin combinations ✓ Edge-detection logic to prevent double-counting held switches

Clock & Input Conditioning

  • Divided the 100 MHz Basys 3 clock down to ~100 Hz for responsive, debounced switch handling.
  • Implemented rising-edge detection on each coin switch so holding a switch down only registers as a single coin insertion.

FSM Design

  • Modeled running totals from 0¢ to 20¢ as states S0–S4, with dedicated dispense states (S5–S9) handling exact payment and every over-payment/change combination up to 45¢.
  • Used Moore-style outputs so dispense and return signals are driven purely by current state, holding until all coin switches are released.

Verification

  • Wrote a self-checking testbench that reads expected outputs from a test-vector file and flags mismatches automatically.
  • Covered 10 payment scenarios — from exact 5-nickel payment to 45¢ over-payment requiring two dimes back — before loading onto hardware.
  • Confirmed correct dispense/return behavior live on the Basys 3, matching LED indicators to expected states (e.g., quarter → immediate dispense, nickel + quarter → dispense + return nickel).

U3

SplitSeez — Receipt-Splitting Web App

Team capstone-style project · QA Engineer

Node.js React MongoDB OCR Team of 6

A full-stack application that scans a restaurant receipt with OCR and automatically splits the bill across a group — by item or by percentage. I served as a QA Engineer on a 6-person team, designing and running the test plan that shaped the final bug list going into demo.

RoleQuality Assurance Engineer (with Sakina Rahman), alongside PM, product marketing, system architecture, and development leads
StackReact frontend, Node.js/Express backend, MongoDB (Mongoose) storage, Tesseract OCR for receipt scanning, JWT-based auth middleware
ScopeUser accounts, group/event management, itemized & percentage-based bill splitting, sub-5-second response time target, 2–10+ person groups
✓ Verified bill-splitting math for both item and percentage modes ✓ Found & tracked 5 high-severity bugs pre-demo ⚠ Two known issues shipped open (event participant removal)

Testing Approach

  • Ran primarily manual testing across core features: registration, duplicate-email handling, concurrent login, event membership, and both bill-splitting modes.
  • Logged results in a structured pass/fail matrix with reproduction notes for every failure.

Bugs Found & Resolution

  • Identified a high-severity CORS misconfiguration that blocked the frontend from reaching backend endpoints entirely — fixed by adding proper CORS headers.
  • Caught missing validation on itemized splits (no way to delete a line item) and on percentage splits (totals not required to sum to 100%) — both patched before demo.
  • Flagged an "owed amount not updating" bug traced to the participating-events fetch, fixed by correcting the response payload.
  • Two issues — participant removal not syncing to receipt data, and duplicate active logins — remained open at demo time and were documented as known limitations.

Lessons Learned

  • Small bugs (like a missing CORS header) revealed larger architectural gaps, which pushed the team toward separating receipt and event data more cleanly.
  • OCR accuracy and remote team coordination across schedules were the two biggest recurring challenges.
